# 视觉质量门禁

本文件用于防止“脚本通过但页面很丑”。它把“高级感、舒服、优雅”拆成可执行、可截图复核、可返工的规则。

## 核心原则

- 脚本通过不等于完成。
- Chrome 技术验收通过不等于完成。
- 必须基于截图做视觉质量评分。
- 视觉质量总分低于 85 分不得交付。
- 任一关键单项低于 70 分必须返工。
- 如页面只是“灰背景 + 白卡片 + 蓝链接 + 表格边框”，不得声明完成。

## 评分表

总分 100 分。每次完成声明必须列出分数。

| 项目 | 分值 | 合格表现 |
|---|---:|---|
| 内容身份清晰度 | 15 | 第一屏能判断页面类型；标题、元信息、目录、正文关系明确。 |
| 层级舒适度 | 20 | H1/H2/H3 差异合理；一屏内没有多个元素争抢主视觉；章节节奏清楚。 |
| 信息密度 | 20 | 正文行宽适中；表格可扫读；列表不松散、不拥挤。 |
| 组件辨识度 | 15 | 目录、表格、代码、元信息、提示、风险等组件有不同处理。 |
| 克制与一致性 | 15 | 色彩、边框、阴影、圆角一致；无廉价装饰和随机样式。 |
| 截图观感 | 15 | 第一屏、表格区、长文区、深色模式都像专业交付物。 |

## 负面清单

出现以下任一情况，不能用“审美偏好差异”解释，必须返工：

- 整页白卡片堆叠。
- 所有 section 使用同一种视觉样式。
- 只加灰背景、白容器、蓝链接就声明完成。
- 目录退化成普通链接长串。
- 表格只加边框，未处理列宽、密度、表头、滚动和数字阅读。
- H1 很大但正文层级没有处理。
- 第一屏只有巨大标题，没有元信息、目录、摘要或正文入口。
- 页面明显像 Word、Markdown 或浏览器默认导出。
- 为了“好看”引入营销 hero、大渐变、装饰背景图。
- 为了“好看”改写、删减、重排、合并用户内容。

## 截图复审

Chrome DevTools MCP 验收必须至少保留以下截图证据：

- 第一屏浅色截图。
- 第一个表格密集区截图。
- 一个长文段落区截图。
- 第一屏深色截图。

每张截图都要检查：

- 是否存在整页空泛或过度拥挤。
- 是否存在标题、正文、表格层级混乱。
- 是否存在所有模块长得一样。
- 是否存在目录不可扫读。
- 是否存在表格可读性差。
- 是否存在廉价装饰或默认导出感。

## 二次优化协议

如果视觉质量总分低于 85，或任一关键单项低于 70：

1. 明确失败项。
2. 说明失败原因。
3. 在不改变内容结构的前提下二次优化 UI。
4. 重跑静态检查。
5. 重跑 Chrome DevTools MCP 技术验收。
6. 重新截图并重新评分。

完成声明不得只写“已优化”。必须写明视觉评分和是否发生二次优化。
